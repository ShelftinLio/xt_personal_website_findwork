## 核心组件

### 1. **GLMService** (`src/services/glmService.ts`)

**职责**: 封装与智谱 AI API 的通信

**核心代码解析**:
```typescript
async chat(messages: Message[], model: string = 'glm-4-flash'): Promise<string> {
  // 1. 发送 HTTP POST 请求到智谱 AI
  const response = await axios.post(
    'https://open.bigmodel.cn/api/paas/v4/chat/completions',
    {
      model: 'glm-4-flash',           // 使用的模型
      messages: messages,              // 对话历史
      temperature: 0.7,                // 创造性参数 (0-1)
      top_p: 0.9,                      // 核采样参数
      max_tokens: 2000,                // 最大输出长度
    },
    {
      headers: {
        'Authorization': `Bearer ${this.apiKey}`  // API 认证
      }
    }
  );

  // 2. 返回 AI 生成的文本
  return response.data.choices[0].message.content;
}
```

**关键参数**:
- `temperature`: 控制回答的随机性，越高越有创造性
- `messages`: 对话历史数组，包含 system、user、assistant 角色

---

### 2. **XiaotianSkillService** (`src/services/xiaotianSkillService.ts`)

**职责**: 构建"角色设定"和"知识库"

**核心原理**:

#### **System Prompt（系统提示词）的作用**

```typescript
const systemPrompt = `
# 刘小天个人介绍助手

你是一个专业的个人介绍助手，必须基于以下数据回答：

## 📚 知识来源
${personalInfo}  // ← 你的完整个人信息

## 🎨 回答风格
1. 使用 Markdown 格式
2. 第一人称"我"
3. 专业且亲和的语气

## ⚠️ 重要约束
- 不得编造信息
- 必须基于数据源回答
`;
```

**为什么 System Prompt 如此重要？**

在发送给 AI 的 `messages` 数组中：
```typescript
[
  {
    role: 'system',      // ← 系统提示词：定义 AI 的"角色"和"知识"
    content: systemPrompt
  },
  {
    role: 'user',        // ← 用户的问题
    content: "介绍一下你的教育背景"
  }
]
```

AI 模型会：
1. **首先阅读 System Prompt** → 理解"我是谁，我知道什么"
2. **然后分析 User Question** → 判断用户想了解什么
3. **基于 System Prompt 中的数据** → 生成准确回答

---

### 3. **MarkdownRenderer** (`src/components/MarkdownRenderer.tsx`)

**职责**: 将 AI 返回的 Markdown 文本转换为格式化的 HTML

**示例**:
```
输入 (AI 返回的纯文本):

- 澳门大学物联网硕士
- 西南交通大学本科

输出 (渲染后的 HTML):

  <li>澳门大学物联网硕士</li>
  <li>西南交通大学本科</li>
</ul>
```

## 关键技术

### 1. **Prompt Engineering（提示词工程）**

这是整个系统的核心技术！

**如何让 AI "知道"你的信息？**

不是训练模型（那需要微调），而是通过 **System Prompt 注入知识**：

```typescript
// ❌ 错误做法：让 AI "自由发挥"
const badPrompt = "你是刘小天的助手";

// ✅ 正确做法：提供完整数据
const goodPrompt = `
你是刘小天的助手，必须基于以下信息回答：

# 教育背景
- 澳门大学物联网硕士，GPA 3.54/4
- 西南交通大学本科

# 实习经历
- 蔚来：AI智能客服，覆盖90%场景
- 京东：热搜榜优化
...

约束：不得编造信息，必须基于以上数据回答
`;
```

**为什么这有效？**
- 大语言模型（LLM）有强大的"上下文理解能力"
- System Prompt 中的信息会被视为"已知事实"
- AI 会基于这些事实回答，而不是产生幻觉

---

### 2. **大语言模型的"上下文窗口"**

**问题**: System Prompt 有长度限制吗？

**答案**: 有！GLM-4-Flash 的上下文窗口约为 **128K tokens**（约 10 万汉字）

**计算**:
```
你的个人信息: ~2,000 字符 ≈ 1,000 tokens
用户问题: ~50 字符 ≈ 25 tokens
AI 回答: ~500 字符 ≈ 250 tokens

总计: ~1,275 tokens ≪ 128,000 tokens ✅
```

**结论**: 你的数据远小于上下文限制，可以轻松放入！

---

### 3. **为什么不使用 RAG（检索增强生成）？**

**传统 RAG 架构**:
```
用户问题 → 向量化 → 检索向量数据库 → 找到相关段落 → 输入 LLM → 生成回答
```

**你的系统（Prompt-based）**:
```
用户问题 + System Prompt(包含全部数据) → LLM → 生成回答
```

**对比**:

| 特性 | 传统 RAG | 你的系统 |
|------|---------|---------|
| **数据大小** | 适合海量数据（百万级文档） | 适合小规模数据（<100KB） |
| **实现复杂度** | 高（需要向量数据库、嵌入模型） | 低（只需提示词） |
| **准确性** | 中等（依赖检索质量） | 高（全部数据在上下文中） |
| **成本** | 高（多次 API 调用） | 低（单次 API 调用） |
| **实时性** | 需要更新向量库 | 更新提示词即可 |

**你的系统优势**:
- 数据量小（个人信息 <10KB）
- 实现简单（无需数据库）
- 回答准确（所有信息都在上下文中）

**何时需要升级到 RAG？**
- 当数据量 > 100KB（比如整个 GitHub 仓库）
- 当需要实时更新（比如新闻文章）
- 当需要精确引用（比如法律文档）

### 4. **Markdown 渲染原理**

**AI 为什么会返回 Markdown？**

因为我们在 System Prompt 中明确要求：
```typescript
请始终用 Markdown 格式回答，使用 #、##、###、- 等标记
```

**渲染流程**:
```
AI 输出 (纯文本):
"# 标题\n\n- 列表项1\n- 列表项2"
        ↓
MarkdownRenderer 解析
        ↓
React 组件树:
<h1>标题</h1>
<ul>
  <li>列表项1</li>
  <li>列表项2</li>
</ul>
        ↓
应用 CSS 样式:
<h1 class="text-2xl font-bold text-white">标题</h1>
...
```

## 总结

### 核心原理

1. **不是训练模型**，而是通过 **System Prompt 注入知识**
2. **不是数据库检索**，而是将数据放入 **LLM 上下文窗口**
3. **不是魔法**，而是 **大语言模型的自然语言理解能力**

### 为什么有效？

- GLM-4-Flash 是 **128K tokens** 的大上下文模型
- 你的个人信息 **<2KB**，远小于限制
- System Prompt 作为"已知事实"，AI 会基于此回答

### 技术优势

✅ **简单**: 无需向量数据库、无需模型微调
✅ **准确**: 所有数据都在上下文中，不会遗漏
✅ **实时**: 更新数据只需修改 System Prompt
✅ **低成本**: 单次 API 调用，响应快速
